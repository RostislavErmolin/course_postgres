# Задание 4: аутентификация и роли
## Описание

На этом занятии мы существенно усилим безопасность нашего приложения. Действовать мы будем с двух направлений:
с точки зрения развертывания и с точки зрения логики самого приложения. 

### Безопасность в развертывании

Что означает безопасность с точки зрения развертывания приложения? А это именно та система контроля доступа postgres,
которую мы разбирали на лекции. 

В нашей системе, в зависимости от задач, могут быть разные типы акторов: catalog manager, 
sales manager, inventory manager, worker. Логично ограничить данные и операции над ними только теми, которые нужны 
данному актору. Это можно сделать с помощью системы контроля доступа postgres, определив соответствующие роли. Затем
при развертывании креденшиалы нужной роли должны использоваться для установления сеанса с postgres.

Например, если мы разворачиваем
приложение для кого-то из отдела продаж, то использование роли sales_manager для создания сеанса postgres, позволит
задать права необходимые только отделу продаж. В частности, сотруднику отдела продаж никогда не понадобится информация
о передвижении товаров между складами, поэтому можно полностью запретить доступ к соответствующей таблице. 
Однако отдел продаж полностью отвечает за заказы, и поэтому любой сотрудник отдела
должен иметь возможность совершать любые операции над ними.

### Аутентификация и авторизация

С другой стороны, подключение к БД из-под определенного пользователя вовсе не означает, что доступ был санкционирован.
Для того, чтоб убедиться в том, что конкретный сотрудник имеет полномочия делать что-то, нам нужно добавить 
аутентификацию и авторизацию в наше приложение. Аутентификация определяет, _кто_ совершает действия. Авторизация 
определяет, какие действия имеет право совершать этот кто-то.

По сути наша задача добавить в приложение форму ввода логина и пароля. Под капотом это потребует создания таблицы для
хранения пользователей. Авторизацию проще всего сделать с помощью ролей. Не путать с ролями postgres! Название этих ролей
могут даже совпадать с ролями postgres, но они работают на разных уровнях. Для простоты будем считать, что у каждого пользователя
может быть только одна роль.

Таким образом таблица с пользователями users должна хранить следующую информацию:
* username
* password
* role -- одно из возможных значений (catalog_manager или sales manager)

Для простоты мы будем считать, что вся часть связанная с auth не относится к нашему приложению и домену -- она 
управляется извне, что в целом логично и похоже на реальный сценарий.

Добавление и управление ролями postgres не должно осуществляться через миграции приложения. Это должно быть сделано
из-под супер пользователя, как часть разделения отвественности в контексте безопасности. 

Но что же делать с миграциями и правами? Проблема в том, что наши миграции запускаются из-под роли app_user. Поэтому
всеми объектами владеет app_user. Мы могли бы передать владение в миграции, но у этого подхода есть ряд
неочевидных недостатков. Во-первых, в down миграции мы не сможем передать владение обратно (кстати, почему?). Во-вторых,
это размазывание владения по ролям может быть не очень удобным и небезопасным. Более удобная ментальная модель -- есть
одна более "главная" роль, которая владеет всеми объектами и раздает права. Она же может и отобрать их, если надо.
Можно сказать, что это соотвествует концепции уровней доступа. app_user -- не суперпользователь, но обладает повышенными
привилегиями. А роли catalog_mnanager и sales_manager -- уже пользовательские, и из-под них можно создавать сеанс в 
приложении.

Поэтому в наших миграциях мы должны явно передавать владение нужной роли на все уже созданные и будущие объекты. Создайте
миграцию, которая распределит права согласно заданию для ролей catalog_manager и sales_manager. 

Для того, чтоб проверить,
поведение нашего приложения из-под разных ролей, необходимо сделать следующее
* добавить разные конфигурации запуска -- по одной на роль, а в этих конфигурациях задать параметры подключения через env перменные
* модифицировать код так, чтоб параметры подключения получать из env переменных

Пример получения параметров из env переменных

## Auth

Хоть наше приложение и учебное, но даже в учебном приложении не стоит хранить пароли в явном виде. Тем более,
postgres имеет встроенное криптографическое расширение pgcrypto, поддерживающее современные стандарты безопасности. Оно
без проблем позволит преобразовать пароль в хэш, сохранив возможность проверять правильность пароля.

Чтобы начать работать с pgcrypto, необходимо установить расширение. Делается это командой
```sql
CREATE EXTENSION IF NOT EXISTS pgcrypto;
```

Для того чтобы сохранить пароль в виде хэша, нужно использовать функцию _crypt('пароль', gen_salt('bf))_. _gen_salt('bf')_ --
означает, что мы будем использовать алгоритм Blowfish для хэширования. gen_salt не только генерирует соль, но и 
оркестрирует функцию crypt. При проверке корректности пароля нужно вызвать проверку pass_hash = crypt('input', pass_hash).
Здесь pass_hash -- имя поля в таблице, где хранится хэш пароля. Подробнее можно прочесть [тут](https://postgrespro.ru/docs/postgrespro/18/pgcrypto#PGCRYPTO-PASSWORD-HASHING-FUNCS).

Например, вставим пользователя командой
```sql
insert into auth.users (username, password, role) VALUES ('cat_man', crypt('pass', gen_salt('bf')), 'catalog_manager');
```

Тогда проверка пароля будет выглядеть так
```sql
select password = crypt('pass', password) from auth.users where username = 'cat_man';
```

Все действия по созданию таблицы users и вставке пользователей выполняются ВНЕ миграций. Мы считаем, что никакого
отношения к нашему приложению аутентификация не имеет. Она находится в другом домене. Поэтому можно выполнять
все команды из-под супер пользователя. Не забудьте выдать корректные права на схему auth и таблицу users.

```python
from typing import Final, Sequence

from prompt_toolkit import prompt

from console import console
from users import User, find_user_by_login_and_pass

ROLE_CATALOG_MANAGER: Final[str] = "catalog_manager"
ROLE_SALES_MANAGER: Final[str] = "sales_manager"

ALL_ROLES: Final[Sequence[str]] = (
    ROLE_SALES_MANAGER,
    ROLE_CATALOG_MANAGER,
)

_USER: User | None = None


def login(username: str | None = None, password: str | None = None) -> None:
    global _USER
    # Try CLI auth if both provided
    if username and password:
        user = find_user_by_login_and_pass(username, password)
        if user:
            if user.role not in ALL_ROLES:
                raise ValueError("Invalid user role")
            console.print(
                f"\n[green]✓ Вход выполнен как {user.username} ({user.role})[/green]\n"
            )
            _USER = user
            return

        console.print("\n[red]✗ Неверные учетные данные из CLI[/red]")
        console.print("[yellow]Попробуйте ввести вручную:[/yellow]\n")

    # Interactive prompt
    console.print("\n[bold cyan]═══════════════════════════════════════[/bold cyan]")
    console.print("[bold cyan]   Вход в систему[/bold cyan]")
    console.print("[bold cyan]═══════════════════════════════════════[/bold cyan]\n")

    while True:
        username = prompt("Имя пользователя: ").strip()
        password = prompt("Пароль: ", is_password=True).strip()
        user = find_user_by_login_and_pass(username, password)

        if user:
            if user.role not in ALL_ROLES:
                raise ValueError("Invalid user role")
            console.print(
                f"\n[green]✓ Вход выполнен как {user.username} ({user.role})[/green]\n"
            )
            _USER = user
            return

        console.print("\n[red]✗ Неверное имя пользователя или пароль[/red]\n")


def auth_user() -> User:
    """
    Вернет аутентифицированного юзера
    """
    if _USER is None:
        raise RuntimeError("Not authenticated")
    return _USER
```

Вам нужно расширить его, добавив поле allowed_roles. Будет как то так

```python
@dataclass(frozen=True)
class Command:
    text: str
    handler: Callable[..., None]
    description: str
    category: str
    allowed_roles: Sequence[str]
    args: Sequence[str] = field(default_factory=tuple)
```

Ниже добавляем в аргумент allowed_roles в декоратор

```python
def command(text: str, description: str, category: str, allowed_roles: Sequence[str]):
    """
    Декоратор для регистрации команд.
    Автоматически извлекает аргументы из сигнатуры функции.
    """

    def decorator(func: Callable[..., None]) -> Callable[..., None]:
        sig = inspect.signature(func)
        args = tuple(sig.parameters.keys())

        cmd = Command(
            text=text,
            handler=func,
            description=description,
            category=category,
            allowed_roles=allowed_roles, # вот это добавили
            args=args,
        )
        _COMMANDS_REGISTRY.append(cmd)
        return func

    return decorator
```

И последнее, меняем функцию get_commmands, чтобы работал фильтр по ролям

```python
def get_commands() -> Sequence[Command]:
    """Возвращает список всех зарегистрированных команд."""
    user = auth_user()
    cmds = [c for c in _COMMANDS_REGISTRY if user.role in c.allowed_roles]
    return cmds
```
