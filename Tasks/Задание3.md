# Задание 3: проектирование БД
## Описание

На прошлом занятии мы добавили товары, категории и склады, а также добавили возможность совершать CRUD операции над ними.
Теперь настало время двигаться дальше. Сегодня мы добавим в нашу систему заказы. Но прежде всего, нам надо
познакомиться с такой концепцией как _миграции_.

### Миграции

Миграции -- это механизм, обеспечивающий хранение схемы (структуры) базы данных в коде. По сути, это набор упорядоченных
скриптов, которые последовательно изменяют структуру БД, и, как правило, хранятся в одном репозитории вместе
с кодом приложения.

Зачем это нужно:

1. Воспроизводимость -- любой разработчик поднимает идентичную схему БД
2. История изменений -- видно кто, когда и зачем менял структуру
3. Откат -- можно вернуться к предыдущей версии схемы
4. Синхронизация окружений -- dev/stage/prod имеют одинаковую структуру
5. Автоматизация деплоя -- изменения схемы накатываются автоматически

И пусть не все эти свойства нам пригодятся в учебном проекте, тем не менее, далее мы будем вносить изменения в
структуру БД только через миграции.

### Заказы

Заказы -- фиксируют факт покупки товаров пользователем, а также указывают на то, какие именно товары и по какой
цене были приобретены. Используя информацию в заказе, магазины должны обеспечить доставку купленных
товаров пользователю.

Мы ограничимся лишь очень небольшой частью данных, которые в реальной жизни представляли
бы данный кусок предметной области. И мы максимально упростим этот набор данных.
Вот, что нас интересуют:
* status -- статус заказа, может быть 'unpublished', 'new', 'processing', 'pending', 'packing', 'shipped'. По-умолчанию 'unpublished'. Нельзя поменять при редактировании заказа.
* total_amount -- общая сумма заказа, должна совпадать с суммой всех товаров в заказе с учетом их количества
* created_at -- когда заказ был создан
* warehouse_id -- идентификатор склада, с которого ожидается отгрузка заказа

И, наконец, нам нужно понимать, а какие именно товары, по какой цене (price) и в каком количестве (quantity)
были куплены в рамках заказа.

### Товары

Несмотря на то, что мы уже добавили товары на предыдущем занятии, на этом занятии нам
необходимо убедиться, что связь между товаром и категорией оформлена через внешний ключ. Таким
образом, postgres поможет нам обеспечить ссылочную целостность.

## Задачи
1. С помощью приложения drawio нарисовать ER диаграмму для складов, товаров, категорий товаров и заказов.
2. Пересоздать существующую структуру БД с помощью миграции. Использовать ограничения ссылочной целостности между товарами и категориями.
3. Написать миграции для создания таблиц orders и order_items, чтоб хранить заказы.
   Заказы должны храниться в отдельной схеме sales (не catalog).
4. Сгенерировать ER-диаграмму для актуальной БД, используя CASE-средство DBeaver. Сравнить две ER-диаграммы.
5. В приложении добавить CRUD операции для управления заказами.
6. Добавить команду publish order. Она меняет статус заказа с unpublished на new. После этого заказ нельзя редактировать и удалять. 

## Инструкции

### ER-диаграмма

Первая задача может быть сделана с помощью материалов лекции:) Примитивы для рисования ER-диаграммы в drawio можно
обнаружить в панели с левой стороны в разделе Entity Relation. Также можно глянуть материал по ссылкам
https://drawio-app.com/blog/entity-relationship-diagrams-with-draw-io/ и https://drawio-app.com/blog/entity-relationship-diagrams-with-draw-io/.
Обратите внимание, что мы рисуем диаграмму в Crow's foot нотации.

### Миграции alembic

Миграции будем писать и выполнять с помощью библиотеки [alembic](https://alembic.sqlalchemy.org/en/latest/).
Конфигурация alembic находится в файле alembic.ini. Единственный параметр, который нас там интересует, это
sqlalchemy.url. Его надо поправить соответствующим образом, задав корректное имя пользователя (app_user) и пароль.

Чтоб создать миграцию, выполните команду
```shell
alembic revision -m "название миграции"
```
где вместо "название миграции" стоит кратко описать, что данная миграция будет делать. Например,
"create products and warehouses tables". После выполнения данной команды в директории `alembic/versions`
появится файл с именем вида `55663709e03f_create_products_and_warehouses_tables.py`. В нем находится скрипт,
который отвечает за выполнение конкретной миграции. С ним ничего делать не нужно. Но не лишним будет
посмотреть на его содержание. Все дальнейшие миграции будут похожи. Единственное отличие будет в
названии ревизии
```python
revision: str = '55663709e03f'
```
Оно-то нас и интересует.

Поскольку это первая миграция, то в директории `alembic` нужно создать поддиректорию `sql`. Именно в ней будут
располагаться все sql скрипты для изменения схемы БД. В поддиректории `sql` нужно создать поддиректорию
с названием ревизии, а в ней два пустых файла: up.sql и down.sql. В итоге должна получиться подобная структура
```
alembic/55663709e03f/up.sql
alembic/55663709e03f/down.sql
```
В файле up.sql мы будем писать sql команды для дальнейшего изменения схемы БД, а в файле down.sql будет скрипт,
который откатит эти изменения до предыдущей версии.

Еще одна полезная команда
```shell
alembic history
```
Она покажет порядок миграций и то, в каком состоянии находится БД в данный момент.

### Первая миграция

В данный момент, у нас уже есть таблицы для продуктов и складов. Если вы почувствуете, что вам
надо пересоздать все таблицы с нуля, (например, чтоб добавить внешний ключ), не стесняйтесь
делать это. Потому что именно сегодня мы имеем дело с нашей первой миграцией, и здесь можно все:)
Дальше такого уже не будет.

Не забудьте, что в миграции должны быть отражены также и операции со схемами.

Может быть полезным увидеть DDL код, который УЖЕ использовался для создания этих таблиц на прошлом занятии. Существует
множество способов это сделать, включая различные CASE средства. На данном занятии, однако, мы попробуем
обойтись утилитой pg_dump https://postgrespro.ru/docs/postgrespro/18/app-pgdump. Она позволяет экспортировать
базу данных. У нее есть масса параметров для настройки экспорта. Нас
интересует только схема, поэтому в экспорте нужно указать параметр --schema-only. Для экспорта конкретной
таблицы команда будет выглядеть так
```shell
pg_dump -t 'schema-name.table-name' -U user-name --schema-only database-name
```
Или можно экспортировать все таблицы из конкретной схемы
```shell
pg_dump -t 'schema-name.*' -U user-name --schema-only database-name
```
Также, можно указать,
например параметр --no-owner, чтоб исключить из экспорта информацию о владельце объекта.
Результат выведется в stdout. Его можно сохранить в файл с помощью оператора `>`, например
```shell
pg_dump -t 'schema-name.*' -U user-name --schema-only database-name > dump.sql
```
Восстановить дамп можно с помощью утилиты pg_restore, но это выходит за пределы наших задач на сегодня.

Итак, после того, как мы получили DDL существующих таблиц, мы можем использовать его в up.sql, для пересоздания
структуры с помощью миграций. Но не стоит бездумно копировать sql код! Обратите внимание, что результат вывода
pg_dump может быть более подробным, чем те команды, которые вы использовали на прошлой лекции. Например, там
отдельно приведен код для создания последовательностей. Вы же можете избежать этого, используя псевдотип serial.

Чтобы накатить миграцию, нужно выполнить команду
```shell
alembic upgrade head
```
А чтобы откатиться до предыдущей миграции, выполните
```shell
alembic downgrade -1
```
Здесь -1 показывает, на сколько миграций нужно откатываться.

При попытке накатить первую миграцию, вас может настигнуть ошибка
```
(psycopg.errors.InsufficientPrivilege) permission denied for schema public
```
Это связано с тем, что alembic создает свою таблицу для хранения текущей ревизии миграции. И делается это в схеме
public. Но public по-умолчанию не доступна всем пользователям для создания объектов.
Мы же выполняем миграции из-под пользователя app_user. Мы могли бы переключиться на пользователя postgres, но
тогда все объекты, которые мы будем создавать в рамках миграций, будут принадлежать суперпользователю.

Ситуацию можно исправить, дав пользователю app_user права на создание объектов в схеме паблик
```sql
GRANT CREATE ON SCHEMA public TO app_user;
```

### DBeaver
DBeaver -- это CASE-средство, которое существенно облегчает работу с СУБД. Поставляется в виде десктопного приложения.
DBeaver предоставляет (но не ограничивается) следующие возможности:
* навигация по схемам и объектам СУБД
* визуализация данных в таблицах
* построение ER диаграмм для существующей БД
* написание SQL запросов с автодополнением
* генерация DDL
* экспорт/импорт данных

Запустите DBeaver. В левой панели главного окна во вкладке Database Navigator создайте подключение к postgres, щелкнув
ПКМ и выбрав Create -> Connection. Используйте креденшиалы суперпользователя postgres -- так мы можем быть уверены, что нам будут
доступны все объекты. Не забудьте выбрать БД inventorydb при создании подключения.

Проверьте, что соединение установлено, и что вам видны все схемы. Далее вы можете визуализировать БД.
Для этого в левом нижнем углу в панели Files, ПКМ по Diagrams и далее Create New Diagram. Укажите только те схемы, которые
представляют интерес: catalog и sales. Вы должны увидеть ER диаграмму,
отражающую текущее состояние БД. Щелкнув ПКМ по ней, можно поменять нотацию на знакомую вам Crow's Foot (по-умолчанию IDEF1X),
а также включить отображение схем на элементах диаграммы, выбрав пункт Show qualified names в меню View styles.

Также можно легко увидеть данные в конкретных таблицах. Достаточно два раза кликнуть на таблицу.

Отдельно стоит упомянуть возможность писать SQL команды, создав SQL script через вкладку SQL Editor. DBeaver поможет
дополнить sql команды в соответствии с текущими объектами БД. Чтоб выполнить SQL команду, нажмите ctrl + Enter.

DBeaver позволяет визуализировать всю внутреннюю структуру кластера postgres. Рекомендуется потратить какое-то время,
и потыкать на разные элементы, чтоб оценить возможности этого CASE-средства.

В любом случае, этот инструмент теперь доступен для вас. Его можно использовать во всех остальных заданиях.

### Заказы

Нам нужно управлять 2 сущностями, одна из которых (order_item) является слабой. Поэтому, помимо обычных CRUD команд
list orders, show/add/edit/delete order <order_id>, нужно будет также добавить еще и команды для order_item:
add/edit/delete order_item <order_id>. Обратите внимание, что параметром в данной команде также является order_id.

Для команд edit order_item и delete order_item следует дать возможность пользователю выбрать, над каким
элементом, совершить операцию. Сделать это можно с помощью функции выбора prompt_toolkit
[choice](https://python-prompt-toolkit.readthedocs.io/en/stable/pages/asking_for_a_choice.html),
с которой вы уже могли столкнуться при реализации выбора категории продукта.

Выбор товаров для добавления order_item должен учитывать существующие товары в заказе, и **не предлагать выбрать из тех, что уже присутствуют**.

При выборе продукта в order_item следует дать возможность пользователю выбрать продукт с помощью автокомплита.
Все-таки потенциально продуктов может быть много, поэтому выбирать из списка не совсем корректно с точки зрения UX.

При добавлении заказа, ожидается, что пользователю также предложат сразу добавить order_items. Хотя и после
создания заказа можно добавить дополнительные товары с помощью add order_item. После каждого
добавления товара в заказ, нужно спрашивать пользователя, хочет ли он добавить что-то еще.
Таким образом, обеспечивается интерактивное добавление нескольких товаров сразу.

Поле created_at не нужно давать заполнять пользователю при создании или редактировании. Оно должно
задаваться по умолчанию при создании заказа.
