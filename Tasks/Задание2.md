# Задание 2. Разработка проекта: каталог товаров и складов
## Описание
Представьте небольшую сеть магазинов электроники. У них есть центральный склад и несколько торговых точек.
Каждый день:

- Товары поступают от поставщиков на центральный склад
- Менеджеры создают заказы для клиентов
- Кладовщики (рабочие) собирают и отправляют заказы
- Товары перемещаются между складами для балансировки остатков

Наша задача — создать REPL (read-eval-print-loop) приложение для автоматизации этих процессов.

Что будет в системе:

- Каталог товаров — что продаём (ноутбуки, мыши, мониторы)
- Склады — где хранится (центральный склад, магазины)
- Остатки — сколько товара на каждом складе
- Заказы — что заказали клиенты
- Движения — перемещения товаров (приёмка, отгрузка, перемещение)

На этом занятии мы создаём фундамент: категории товаров, товары и склады, и даем возможность выполнять CRUD
(create, read, update, delete) операции над ними.

Данные, которые нам нужно хранить для продуктов:
- sku (stock keeping unit) -- уникальная строка, артикул, не пустой, не более 30 символов
- name -- имя товара, не пустое
- price -- цена, не пустое

Кроме того каждый товар должен принадлежать одной из категорий. Категория представляет собой сущность
с единственным полем name, обозначающим название данной категории. Поэтому принадлежность товара к категории нужно
оформить в виде ссылки на запись из таблицы с категориями.

Данные, описывающие склад:
- city -- город, в котором находится склад, не пустой
- address -- адрес склада, не пустое
- label -- дополнительная метка, помогающая идентифицировать склад (например, "На Дыбенко" или "На Пионерской"), может быть пустым (nullable)
- is_central -- булеан флаг, может быть true у единственного склада

Учтите, что центральный склад должен быть хотя бы один и только один! Операции редактирования и удаления должны учитывать это
требование. Если добавляется самый первый склад, то он делается по-умолчанию центральным.

Помимо данных, описанных выше, склады, товары и категории должны обладать уникальным идентификатором.

## Задачи
1. Создать базу данных `inventorydb` в postgres. Создать пользователя `app_user` с паролем. Создать схему `catalog`,
   в которой будет храниться информация о товарах и складах. Пользователь `app_user` должен иметь возможность
   создавать объекты в базе данных `inventorydb` и схеме `catalog`.
2. В схеме `catalog` создать три таблицы `product_categories`, `products` и `warehouses` для хранения информации о категориях,
   товарах и складах соответственно.
3. Используя шаблон (см. инструкцию ниже), создать простое REPL приложение, поддерживающее следующие операции
- list product_categories
- show product_category {id}
- add product_category
- edit product_category {id}
- delete product_category {id}
- list products
- show product {id}
- add product
- edit product {id}
- delete product {id}
- list warehouses
- show warehouse {id}
- add warehouse
- edit warehouse {id}
- delete warehouse {id}

## Инструкции

Задания 1 и 2 можно выполнить используя материал лекции.

Ничего страшного, если связь между категориями и товарами
будет сделана с помощью простого сопоставления id из product_categories и значения поля product_category_id.
Мы оформим правильную связь между сущностями на следующем занятии.

### Развертывание шаблона
1. Для работы над задачей 3, нужно использовать шаблон https://github.com/Feolius/pg-course-template.
Склонируйте или скачайте код шаблона через github GUI. Скопируйте код в рабочую директорию.
2. Запустите pycharm. Откройте в нем рабочую директорию, выбрав в меню `File -> Open`. Pycharm должен предложить
   создать venv. Если этого не произошло, то потребуется сделать это вручную через настройки Project interpreter. **Обязательно выберете python3.10 в качестве базового интерпретатора.** Более свежие версии не будут полноценно поддерживаться pycharm2024.
   
venv также можно создать вручную в терминале.
```shell
python3 -m venv .venv
source venv/bin/activate
pip install -r requirements.txt
```
Возможно потребуется перезапуск pycharm, если venv так и не будет обнаружен. Но надеюсь, это все не потребуется:)

3. Добавьте конфигурацию запуска. Для этого кликните на `Current File` в верхней правой части экрана, и в выпадающем
   меню нажмите `Edit Configurations`. Нажмите `+` в открывшемся окне, и выберете python. В качестве файла скрипта
   укажите `src/main`. **Внимание! В `Modify options` обязательно добавить `Emulate terminal in output console`!**
4. В файле `src/db` поменяйте значения `DB_PASSWORD` на то, которое нужно.
5. Если вы достаточно удачный, то нажатие на кнобку запуска рядом с конфигурацией запуска, либо комбинация `shift + f10`
   запустит шаблонное приложение. Если же не получилось, не отчаивайтесь. Но надо, чтоб получилось в итоге!

### Технические детали шаблона

Ядро REPL реализовано с помощью библиотеки `prompt toolkit`. Документация и примеры находятся
[тут](https://python-prompt-toolkit.readthedocs.io/en/stable/pages/getting_started.html). В шаблоне предложено
использовать `prompt toolkit` исключительно для ввода команд.

Для стилизованного вывода предлагается использовать библиотеку `rich`. С документацией можно познакомиться
[здесь](https://rich.readthedocs.io/en/latest/index.html).

Наша главная цель — это работа с постгрес, поэтому то, на сколько ваше итоговое приложение будет
красивым, в большей степени зависит от вас и от вашего чувства прекрасного.

Что касается интеграции с postgres, то здесь используется одна из самых популярных клиентских библиотек
[psycopg](https://www.psycopg.org/psycopg3/) 3-й версии. Не стесняйтесь заглядывать в документацию, если что-то
непонятно.

### Отладка и логирование
При правильной настройке pycharm, вам доступен дебаггер из коробки. Кнобка со значком жучка запустит приложение
с включенным дебаггером. Можно ставить брейкпоинты и смотреть, что происходит.

Однако иногда может понадобиться распечатать информацию для дебага. Поскольку мы пишем консольное
приложение, не очень удобно писать отладочную информацию в stdout с помощью вызова `print`.
Обратите внимание, что в шаблоне сконфигурирован логгер, который можно использовать для этих целей. При вызове
`logging.info()` или другого аналогичного метода логирования, сообщение попадает в файл `logs/app.log`.

Если вам потребуется залогировать sql запрос, то рекомендую обратить внимание на атрибут
[_query](https://www.psycopg.org/psycopg3/docs/api/cursors.html#psycopg.Cursor._query) объекта cursor библиотеки psycopg.

### Дальнейшие действия

Теперь когда приложение запущено, следует посмотреть файл README и код.

Кто-то уже реализовал необходимую функциональность
для складов, но без учета флага is_central. Ваша задача, используя эту реализацию как образец, сделать то же самое для
продуктов и категорий продуктов, а также добавить поддержку флага is_central.

При реализации операций добавления и редактирования товаров, необходимо обеспечить выбор
категорий из списка существующих. Вы можете сделать это по аналогии с выбором города для
склада (см. city_completer в файле warehouses.py), но лучше воспользоваться
компонентом [choice](https://python-prompt-toolkit.readthedocs.io/en/stable/pages/asking_for_a_choice.html)
из prompt_toolkit. Поскольку список категорий товаров довольно ограничен,
выбор одного из списка представляется более удобным вариантом.
